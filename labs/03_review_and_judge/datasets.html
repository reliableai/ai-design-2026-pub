Review & Judge

From single-shot prompts to self-critique, feedback loops, and systematic evaluation

Accept GitHub Classroom Assignment →

The critique & review pattern

A single-shot prompt gives you one attempt. If the model misses a fact, hallucinates a detail, or misformats the output — that's what you get. The critique and review pattern asks the model to check its own work before delivering a final answer.

The idea is simple: instead of answer once, do answer → critique → revise — all within a single prompt.

Example: self-critique in one prompt

You are a knowledgeable and rigorous assistant.

<task>
{{QUESTION}}
</task>

<process>
1. Draft your answer.
2. Adopt the role of a strict peer reviewer. Check for:
   - Factual accuracy
   - Internal consistency
   - Edge cases and exceptions
   - Clarity for the intended audience
3. Revise once based on the critique.
</process>

Respond ONLY with a valid JSON object, no markdown, no preamble:
{
  "answer": "<your initial answer here>",
  "critical_review": "<your critique here>",
  "answer_v2": "<your revised answer here>"
}

Why this works

  • Forces deliberation. The model must produce an explicit critique before revising. This surfaces errors the model “knows about” but wouldn't catch in a single pass.
  • Transparent reasoning. You get the draft, the critique, and the revision — so you can inspect what changed and why.
  • Cheap to add. It's just prompt text. No extra API calls, no infrastructure. One call, three outputs.

Limitations

  • Same model, same blind spots. The “reviewer” is the same model that made the error. It can catch surface-level mistakes (formatting, obvious contradictions) but it tends to miss the same conceptual errors twice.
  • Token cost. You pay for the draft + critique + revision, roughly 2–3× the tokens of a single answer.
  • Single iteration. The prompt says “revise once.” What if the revision introduced new problems? You'd need another round — but you can't loop inside a single prompt.

The core tension

In-prompt self-critique is better than nothing, but it's bounded by the model's own reasoning in a single forward pass. The model can't go back and re-read its draft with fresh eyes. It can't run the output through a different evaluator. It can't try three approaches and pick the best. For that, you need an external loop.

From in-prompt critique to feedback loops

What if, instead of asking the model to critique itself in one shot, you run the critique as a separate LLM call — and then feed the critique back as input for revision?

Prompt → LLM: generate → LLM: judge → LLM: revise ↺ repeat

This is fundamentally different from in-prompt self-critique:

In-prompt critique External feedback loop
Calls 1 LLM call 2–N calls per iteration
Reviewer Same model, same context Can be a different model or prompt
Iterations Fixed (usually 1) Loop until quality threshold met
Visibility All in one response Each step is a separate, inspectable artifact
Flexibility Locked to one approach Can try multiple variations, pick the best

Example: the review-and-revise prompt

This is not a judge that scores an output. It's the second call in a two-call loop: it receives an original task and a first attempt, then critically reviews it and produces an improved version.

Call 1 — generate the first attempt (any prompt you like):

You are a support-ticket triage assistant.

Given the customer request below, extract:
- intent: the customer's primary goal (short verb phrase)
- symptoms: list of concrete, observable problems

Respond in JSON with keys "intent" and "symptoms".

Customer request:
{{request}}

Call 2 — review and revise (a separate LLM call that receives the output of Call 1):

You are a strict quality reviewer for support-ticket extractions.

Below is a customer support request and an extraction produced by
another system. Your job is to critically review the extraction
and produce a corrected version.

<original_request>
{{request}}
</original_request>

<extraction_to_review>
{{extraction}}
</extraction_to_review>

Review the extraction against the original request:
1. Is the intent accurate? Is it specific enough, or too vague?
2. Are all symptoms from the request captured? Any missed?
3. Are any symptoms hallucinated (not actually in the request)?
4. Is each symptom atomic (one fact per item)?

Then produce a corrected extraction.

Respond ONLY with a valid JSON object:
{
  "review": "your critique of the original extraction",
  "corrected_intent": "the revised intent",
  "corrected_symptoms": ["revised", "symptom", "list"]
}

What makes this different from in-prompt self-critique?

  • Fresh context. The reviewer starts from scratch — it doesn't carry forward the reasoning that led to the original mistakes.
  • Different instructions. Call 1 says “extract.” Call 2 says “find what's wrong and fix it.” Different task framing = different failure modes.
  • Chainable. You can feed the corrected output back into the same review prompt for another round, or pass it to a different reviewer with stricter criteria. The loop is external, so you control when to stop.
  • Observable. Each call is a separate API request with its own input/output. You can log, compare, and debug each step independently.

This two-call pattern is the simplest feedback loop. The lab extends it further: instead of reviewing individual outputs, we run the extraction over hundreds of items, judge them all, and use the aggregate failure patterns to improve the prompt itself.

The extract → judge → improve loop

For the rest of this lab, we apply this pattern to information extraction from support tickets. The loop works like this:

  1. Extract — run a prompt template on a batch of support requests. The LLM produces structured JSON (intent + symptoms) for each request.
  2. Judge — a separate LLM call (the “judge prompt”) rates each extraction for completeness, accuracy, and format. This is the external critic.
  3. Analyze — look at failure patterns. What kinds of requests does the prompt struggle with? What errors keep recurring?
  4. Improve — feed the failures and a best-practices catalog to another LLM call that generates prompt variations — each trying a different technique to fix the identified issues.
  5. Re-test — run each variation on the same data, judge again, and pick the winner.
extract → judge → analyze → improve → re-test ↺

Key insight: the judge is a separate prompt

The judge doesn't see the extraction prompt's instructions — it sees the original request and the extracted output, and rates whether the output faithfully captures what the request says. This separation is what makes the feedback loop powerful: the critic has a different perspective than the generator.

This also means you can swap judges: use an LLM-as-judge for open-ended quality, or compare against ground-truth labels when they exist. Both tell you something different about your prompt's performance.

Why run this at scale?

One example tells you nothing. Ten examples might mislead you. When you run the loop over hundreds of items, failure patterns emerge that you'd never spot from a few samples: the prompt consistently mishandles ambiguous requests, or drops symptoms from long texts, or hallucinates intents for short inputs. That's what motivates the datasets below.

Ticket types & priorities

Why it's useful

  • Multi-paragraph: Symptoms are buried in polite preamble, signatures, and context. The model must separate signal from noise.
  • Subject + body structure: The subject often summarizes the issue, but the body contains the details. Prompts must handle both.
  • Diverse domains: IT infrastructure, software integrations, data analytics, hardware — the model needs to handle domain-specific jargon.
  • Validation metadata: Priority and ticket type serve as proxy labels for judging extraction quality.

Extraction challenge

Many tickets open with "Dear Customer Support, I am writing to report…" before getting to the actual problem. A naive prompt will waste tokens on preamble. The subject line often contains the key information in compressed form.

3. CFPB Consumer Complaints

3.7 million real consumer complaints submitted to the US Consumer Financial Protection Bureau. These are actual people writing about actual problems with actual financial institutions. PII is redacted with XXXX. This is the hardest dataset with labels.

Product categories

Why it's useful

  • Real data: Not synthetic. People ramble, express frustration, mix multiple issues, provide irrelevant context, and make grammatical errors.
  • PII redaction noise: Names, dates, account numbers are replaced with XXXX. The model must extract meaning despite missing entities.
  • Multi-issue complaints: Many narratives describe 2-3 interrelated problems. Testing whether your prompt handles multi-intent extraction.
  • Long text: Average ~1,000 chars, some up to 30,000. Stress-tests prompt instructions when the input is very long.
  • Domain-specific vocabulary: Financial terms (lien release, charge-off, escrow, FDCPA) that a generic prompt won't understand well.

Extraction challenges

  • XXXX redaction makes it hard to tell dates from names from account numbers
  • Complaints often embed forwarded emails, legal citations, or timeline narratives
  • Emotional language ("I am EXTREMELY frustrated") mixes with factual claims
  • Some complaints are near-incomprehensible due to poor grammar or stream-of-consciousness writing

Why it's useful

  • Hardest extraction challenge: Tweets are short, informal, full of abbreviations, typos, and emotional language.
  • Missing context: Customers assume the brand knows what product/service they mean. The model has to infer from minimal cues.
  • No labels: Pure unsupervised extraction — you can only judge quality, not compare to ground truth.
  • Real-world noise: @mentions, hashtags, URLs, emojis, ALL CAPS for emphasis.

Extraction challenges

  • Extremely short — sometimes just "this is broken. fix it." with no other context
  • Multi-turn conversations where the first tweet references a previous interaction
  • Sarcasm, irony, and venting that aren't actual symptom descriptions
  • @mentions and hashtags mixed into the text

The teaching angle

Run the same extraction prompt across all four datasets. Watch the judge scores drop as difficulty increases:

  1. Bitext — scores should be high (4-5). If not, the prompt has fundamental issues.
  2. Tobi-Bueck — scores drop because of verbose text and domain jargon.
  3. CFPB — scores drop further due to real-world noise, multi-issue complaints, and PII redaction.
  4. Twitter — the most challenging: informal, short, no context, no labels to validate against.

Then use the auto-improvement loop to adapt the prompt for each dataset — and see how the optimal prompt differs across domains.
